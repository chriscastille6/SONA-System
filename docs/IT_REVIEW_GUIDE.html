1. Scope and Purpose of PRAMS

The Participant Recruitment and Management System (PRAMS) is a custom, internal web-based platform designed specifically to support and automate research workflows at Nicholls State University.

Primary Scope: Active Research Support

At this point in time, PRAMS is hosted through Dr. Christopher Castille's institutional website (`bayoupal.nicholls.edu/hsirb/`) and is used primarily for Dr. Castille's own research program. In preparing and configuring this tool, Dr. Castille has actively consulted with the current Human Subjects Institutional Review Board (HSIRB) chair, Dr. Alaina Daigle, to ensure compliance, procedural alignment, and utility.

While primarily optimized for Dr. Castille's research, the application's design is fully extensible: if other researchers or instructors on campus wish to utilize PRAMS for their own studies or courses in the future, they are welcome to do so simply by registering an account on the platform.

Principal Automation Goals

PRAMS was developed to make it significantly easier for Dr. Castille to accomplish four key research and administrative objectives in a highly automated, streamlined manner:

1. Study Launch & Collection

Launches active, IRB-approved studies for immediate data collection, managing secure booking timeslots, locations, and participant sign-ups smoothly.

2. Community Engagement Reporting

Simplifies the process of updating the IRB regarding any community engagement activities, public outreach, or localized research events experienced during studies.

3. Lab Insight Sharing

Facilitates the automated sharing of aggregated research insights, performance metrics, and study conclusions directly with the research lab members.

4. End-to-End Automation

Unifies and automates tasks 1–3, reducing manual coordination, paper logs, and human administrative errors in research tracking.

Benefits to Institutional Review Board (IRB) Efficiency

Beyond supporting individual researcher workflows, utilizing PRAMS provides major benefits for the overall efficacy of the Nicholls State University HSIRB. The platform is designed to make it much easier for the IRB to function efficiently and effectively by enabling them to:

  • Increase Throughput: Approve and monitor more studies within a secure, centralized system.
  • Robust Documentation: Maintain complete, immutable, digital records of all protocol reviews, amendments, and approvals.
  • Responsive Communication: Rapidly respond to researchers regarding review decisions, missing documentation, or audit requirements.

2. Security Architecture Dashboard

Role-Based Access Control RBAC

PRAMS enforces strict Role-Based Access Control (RBAC) at the view layer on every single backend route. This ensures that users can only perform actions and access data appropriate to their designated role.

  • Roles: Administrator, IRB Member, Researcher, Instructor, and Participant.
  • Self-Registration: Strictly restricted to the Participant role. All elevated roles are assigned by administrators or via secure, single-use invitation flows.
  • Object-Level Scoping: Queries are scoped to the authenticated user. For example, researchers can only retrieve studies they own, and instructors can only view courses they teach.
IDOR Mitigation Security

Insecure Direct Object Reference (IDOR) vulnerabilities are mitigated by ensuring that database lookups do not rely solely on client-supplied primary keys without ownership validation.

  • Scoped Lookups: Views resolve objects using ownership-scoped queries (e.g., get_object_or_404(Study, id=study_id, researcher=request.user)).
  • 404-for-Unauthorized: To prevent resource enumeration and information disclosure, unauthorized access attempts return a standard 404 Not Found rather than a 403 Forbidden where appropriate.
  • Opaque References: Sensitive public-facing endpoints (such as anonymous booking confirmations) use secure, non-sequential UUIDs and session-scoped confirmation keys.
Password & Auth Security Security

PRAMS leverages Django's robust, industry-standard authentication framework to handle user sessions, password resets, and credentials securely.

  • NIST-Aligned Hashing: Passwords are hashed using Argon2 or PBKDF2 with high iteration counts, protecting against brute-force and rainbow table attacks.
  • Session Security: In production, cookies are configured with HttpOnly, Secure, and SameSite=Lax flags to prevent cross-site scripting (XSS) and cross-site request forgery (CSRF) session hijacking.
  • SSO Readiness: The authentication layer is isolated from authorization logic, allowing seamless integration with campus Single Sign-On (SAML2, OAuth2, or OIDC) in the future.
Secrets & Env Management Security

No sensitive credentials, API keys, or database passwords are stored in the codebase. PRAMS strictly adheres to modern Twelve-Factor App methodology for configuration.

  • Environment Variables: All secrets (including SECRET_KEY, database URLs, email host credentials, and participant export salts) are read from system environment variables.
  • Git Isolation: Local configuration files (such as .env) are explicitly ignored via .gitignore and are never committed to the repository.
  • Production Hardening: The application enforces that a secure, non-default SECRET_KEY is provided when DEBUG = False, preventing startup under insecure default configurations.

3. FERPA & Identity Protection

Protecting student privacy and complying with the Family Educational Rights and Privacy Act (FERPA) is a core design requirement of PRAMS.

Standard Participant Accounts (SSO/Credit Readiness)

For standard studies where students will participate to earn course credit, PRAMS is fully designed to collect basic student identifiers (name, email, student ID) to facilitate automated credit tracking. Note: To maintain an extremely conservative data privacy posture during this launch phase, automated credit allocation is currently deactivated; course credit and incentives are handled manually by the research team outside of the application. However, when standard credit accounts are enabled, strict architectural controls are implemented to isolate and protect this Personally Identifiable Information (PII):

Restricted Access

Only authorized school officials (instructors and administrators) can view student names and IDs for credit verification. Researchers never see student IDs.

Database Isolation

Student account records are stored in separate tables from research response data, preventing direct association between a student's identity and their survey responses.

System-Specific Salting

For research exports, a secure, environment-defined salt (PARTICIPANT_EXPORT_SALT) is used to generate stable, opaque, 32-character hex identifiers via HMAC-SHA256.

Why System-Specific Salting Matters

By using a unique, system-specific salt to hash student identifiers for research exports, PRAMS ensures that the resulting anonymous identifiers cannot be reversed or linked to other external research databases. Even if the same student participates in studies across different systems, their data remains completely un-linkable, eliminating the risk of re-identification via record linkage (the "mosaic effect").

SSO Integration and Anonymity: Architectural Design Comment

The Security/Anonymity Tradeoff: From an IT administration perspective, establishing campus Single Sign-On (SSO) integration is highly desirable because it provides complete **traceability**, identity verification, and central access logging. This is incredibly beneficial for securing the admin, researcher, and instructor panels. However, introducing mandatory SSO authentication can represent a design concern if students wish to sign up for voluntary research studies under a high expectation of complete anonymity.

Co-Existence Strategy (Unified Security + Open Booking): PRAMS handles this trade-off gracefully by separating access controls into two distinct functional channels:

  • SSO-Secured Management (High Traceability): All researchers, administrators, and instructors authenticate via Nicholls SSO. This secures administrative controllers, audit logs, and researcher configurations, providing full corporate traceability where privileged data access occurs.
  • Open Public Signup (Anonymity-Preserving): To preserve participant anonymity for sensitive or cash-compensated research (such as goal-setting trials), student booking endpoints remain decoupled from the SSO authentication flow. These endpoints utilize the Anonymous Booking Module (detailed in Section 4), relying on one-way UUIDs and temporary rate limits rather than institutional user profiles.

Alternative: Pseudo-SSO Signups (Identifiable but Isolated): If campus policies require that all students authenticate via SSO to book slots (preventing public spam), PRAMS can be configured to require SSO login but immediately drop/minimize identity metadata upon successful authentication. At the time of booking, the system would verify the student's active status via SSO, generate a cryptographically salted participant hash, and store only that hash on the reservation, instantly throwing away any username, student ID, or email. This ensures both identity verification (preventing slot-flooding) and database-level participant anonymity.

4. Anonymous Booking Module

For cash-compensated or completely voluntary studies where no course credit is awarded, PRAMS provides a specialized Anonymous Booking module.

This module is crucial for ensuring complete participant anonymity and compliance with strict IRB requirements for sensitive research topics. It allows participants to book a timeslot without creating a user account or providing any identifying information.

Data Attribute Stored in AnonymousSignup? FERPA Education Record? Security & Privacy Controls
Student Name / Email NO No No fields exist in the database table; completely bypassed.
Student ID / Login NO No Zero integration or link to student institutional login or email.
Booking Reference YES No An auto-generated, opaque UUID used solely for cancellation.
Cancellation PIN YES No A random 4-digit PIN, shown only once to the student on booking.
IP Address / User Agent AUDIT ONLY No Stored in append-only AuditLog for rate limiting and abuse protection; never linked to research data.
The AnonymousSignup Database Model

The database schema confirms that the AnonymousSignup table contains absolutely no student PII or foreign keys to the User table. Below is the actual Django model implementation:

class AnonymousSignup(models.Model): """Anonymous timeslot booking — no user account or PII.""" STATUS_CHOICES = [ ('booked', 'Booked'), ('cancelled', 'Cancelled'), ] id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False) timeslot = models.ForeignKey( Timeslot, on_delete=models.CASCADE, related_name='anonymous_signups', ) booking_reference = models.UUIDField( default=uuid.uuid4, unique=True, editable=False, help_text='Reference shown to participant for cancellation', ) cancellation_pin = models.CharField(max_length=4, help_text='4-digit PIN for cancellation') status = models.CharField(max_length=20, choices=STATUS_CHOICES, default='booked', db_index=True) consent_text_version = models.TextField(help_text='Copy of consent text at time of signup') booked_at = models.DateTimeField(auto_now_add=True) cancelled_at = models.DateTimeField(null=True, blank=True)
Anonymous Booking Threat Controls
  • Slot Flooding Prevention: Strict IP-based rate limits are enforced on bookings (default: 5 bookings per hour) to prevent malicious slot exhaustion.
  • PIN Brute Force Protection: IP-based rate limits are enforced on cancellations (default: 15 attempts per 15 minutes), combined with constant-time PIN comparison and generic error messages ("Booking not found or PIN invalid") to prevent enumeration.
  • IDOR Success Page Protection: Confirmation screens are rendered using a one-time session key (anonymous_booking_id) rather than a public, guessable URL.

5. Separation of Research Responses & Contact Info

A common challenge in research is the need to collect survey responses anonymously while still allowing participants to opt-in to receive follow-up materials, study results, or infographics via email.

If the email is stored in the same database table as the survey responses, the responses are no longer anonymous. PRAMS solves this problem through a robust, architectural separation of datasets.

The Response Model

Stores the actual research survey or assessment data submitted by the participant. It is identified solely by an anonymous, random session_id (UUID). It contains absolutely no student names, emails, or student IDs in its payload.

The StudyEmailContact Model

Stores the participant's email address in a completely separate table. While it can store an optional session_id to link the email to a response session (if submitted in the same flow), access to this table is strictly restricted.

Database Implementations

The separation is enforced at the database schema level, preventing accidental or unauthorized linkage of student identities with their research responses:

class Response(models.Model): """Protocol response submitted by a participant.""" id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False) study = models.ForeignKey(Study, on_delete=models.CASCADE, related_name='responses') session_id = models.UUIDField(default=uuid.uuid4, db_index=True, help_text="Anonymous session identifier") payload = models.JSONField(help_text="JSON data submitted from protocol") created_at = models.DateTimeField(auto_now_add=True, db_index=True) ip_address = models.GenericIPAddressField(null=True, blank=True) user_agent = models.TextField(blank=True) class StudyEmailContact(models.Model): """ Optional email signup for sending study infographics. Stored in a separate dataset from Response payloads for IRB/privacy clarity. """ id = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False) study = models.ForeignKey(Study, on_delete=models.CASCADE, related_name='infographic_contacts') email = models.EmailField() session_id = models.UUIDField(null=True, blank=True, db_index=True) created_at = models.DateTimeField(auto_now_add=True, db_index=True)
Security Controls on Linkage
  • Database Permissions: Access to the StudyEmailContact table is restricted. Researchers cannot query or export this table alongside the Response table.
  • No Automatic Merging: Export utilities strictly separate survey response datasets from email contact lists. No merge of these tables is permitted without explicit, audited IRB approval and oversight.
  • Audit Trails: Any cross-table queries or administrative access to the StudyEmailContact table are captured in the system's append-only audit logs.

6. Audit & Review Instructions for IT Security Staff

Nicholls State University IT Security staff can independently audit and verify the security posture of the PRAMS application by following the steps below.

Step 1: Inspect the Repository & Codebase

Access the GitLab repository and review the overall project structure. Focus on the core application directories:

  • apps/studies/: Contains the core models, views, and business logic for studies, signups, and IRB tracking.
  • apps/prams/: Contains the API and catalog single-page application (SPA) views.
  • config/: Contains the global settings and routing configurations.
Step 2: Verify Security Settings

Open config/settings.py and verify the following production-hardening settings:

  • Confirm that DEBUG = False is enforced in production.
  • Verify that security cookies are enabled: SESSION_COOKIE_SECURE = True and CSRF_COOKIE_SECURE = True.
  • Ensure that SECURE_SSL_REDIRECT = True is enabled to force HTTPS.
  • Check that the password hashing backends are aligned with NIST standards (e.g., Argon2 or PBKDF2).
Step 3: Audit Routes and Access Controls

Inspect apps/studies/views.py and apps/prams/views.py to confirm that role-based access controls are active:

  • Verify that all sensitive views are protected by authentication decorators (e.g., @login_required) or custom role-checking mixins.
  • Ensure that no state-changing views bypass CSRF protection (e.g., avoid @csrf_exempt unless strictly token-secured).
  • Confirm that views querying database objects scope their lookups to the authenticated user's role and ownership.
Step 4: Verify Secrets Isolation

Review the environment configuration files:

  • Verify that .env is listed in the root .gitignore file to prevent secrets from being committed.
  • Inspect env.example to confirm that all required secrets (database credentials, email settings, and PARTICIPANT_EXPORT_SALT) are documented without actual values.
Step 5: Audit Database Schemas & Data Separation

Open apps/studies/models.py and review the table definitions:

  • Verify that the AnonymousSignup model contains no fields for student names, emails, or IDs.
  • Confirm that the Response model payload is stored as a JSONField and contains no direct links to student accounts.
  • Verify that the StudyEmailContact model is defined as a separate table and that access to it is restricted.

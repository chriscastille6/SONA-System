1. System Overview

1.1 Purpose and Scope

The SONA Research Participation System is a web-based platform designed to facilitate research participation in psychology and social science studies at Nicholls State University. The system serves three primary functions:

  • Study Management: Researchers can create, manage, and monitor research studies
  • Participant Recruitment: Students can browse, sign up for, and participate in approved studies
  • Credit Tracking: Automated tracking of research participation credits for course requirements

1.2 User Roles

Role Description Access Level
Participants Students enrolled in psychology courses Browse studies, book sessions, view own credits
Researchers Faculty and approved research personnel Create studies, manage sessions, collect data, mark attendance
Instructors Course instructors requiring research participation View enrolled students, monitor credit completion
Administrators System administrators and IRB coordinators Full system access, user management, audit logs

1.3 Workflow Overview

  1. Study Creation: Researcher creates study with IRB approval documentation
  2. IRB Review: Administrator reviews and approves study for participant access
  3. Participant Recruitment: Eligible students browse and sign up for studies
  4. Informed Consent: Participants review and agree to study-specific consent
  5. Participation: Students attend sessions or complete online protocols
  6. Credit Award: Credits automatically awarded upon verified attendance
  7. Data Collection: Research data stored securely with privacy protections

2. Data Collection & Storage

2.1 Types of Data Collected

2.1.1 User Account Data

Collected at Registration:

  • Email address (institutional email required)
  • Name (first and last)
  • User role (participant, researcher, instructor)
  • Password (encrypted with industry-standard hashing)

2.1.2 Participant Profile Data (Optional)

  • Student ID number
  • Date of birth (for age eligibility)
  • Phone number (optional, for appointment reminders)
  • Demographic information (collected only if participant consents for prescreening)

2.1.3 Study Signup Data

  • Timeslot booking information
  • Consent form acceptance (timestamped)
  • Consent form version (stored for audit trail)
  • Attendance status (booked, attended, no-show, cancelled)
  • Booking and attendance timestamps

2.1.4 Research Protocol Data

Important Privacy Feature:

Research protocol responses are stored with anonymous session IDs, not user IDs. This creates a technical separation between participant identity and research data, ensuring that researchers cannot directly link responses to individual participants through the database.

  • Session ID: Randomly generated UUID (not linked to user account)
  • Study ID: Links response to research study
  • Response Payload: JSON-formatted research data (structure defined by researcher)
  • Metadata: Timestamp, IP address (optional), user agent (optional)

2.2 Data Storage

Data Type Storage Location Encryption Access Control
User Accounts PostgreSQL Database At-rest encryption, password hashing (PBKDF2) Role-based access control (RBAC)
Study Signups PostgreSQL Database At-rest encryption Researcher access to own studies only
Protocol Responses PostgreSQL Database (JSON field) At-rest encryption, anonymous session IDs Researcher access to own studies only
Audit Logs PostgreSQL Database At-rest encryption, immutable records Administrator access only

3. Privacy & Security Measures

3.1 Data Anonymization

Multi-Layer Privacy Protection

The system implements a three-tier privacy model for research data:

  1. Session-Based Anonymization: Research responses use random session IDs instead of user IDs
  2. Role-Based Access: Researchers can only access data from their own approved studies
  3. Consent Versioning: Every signup stores the exact consent text shown to participant

3.2 Security Features

Security Layer Implementation Protection Against
Authentication Django authentication with password hashing (PBKDF2-SHA256) Unauthorized access
Authorization Role-based access control (RBAC) Privilege escalation
Transport Security HTTPS/TLS encryption for all data transmission Man-in-the-middle attacks
CSRF Protection Django CSRF tokens on all forms Cross-site request forgery
SQL Injection Protection Django ORM with parameterized queries Database injection attacks
Session Security Secure, HTTP-only session cookies Session hijacking
Audit Logging Immutable logs of sensitive operations Unauthorized changes

3.3 Access Controls

Researcher Access Limitations:

  • Researchers can only view and manage their own studies
  • Cannot access other researchers' study data or participant responses
  • Cannot modify credit transactions (system-controlled)
  • Cannot access complete participant profiles beyond signup information

Participant Access Rights:

  • View only their own bookings and credit balance
  • Cannot access other participants' information
  • Can cancel their own bookings within cancellation window
  • Can request data deletion per FERPA/privacy regulations

4. Informed Consent Process

4.1 Two-Tier Consent Model

Tier 1: System-Level Consent (Account Creation)

When creating an account, participants agree to:

  • General use of the SONA system
  • Storage of contact and enrollment information
  • Email communications about appointments and credits

Tier 2: Study-Specific Consent (At Signup)

Before booking each study, participants must:

  • Read the complete study-specific consent form
  • Explicitly click "I Agree" to consent
  • Consent is timestamped and stored with exact consent text version

4.2 Consent Requirements for Researchers

Researchers must provide:

  1. Study Purpose: Clear description of research objectives
  2. Procedures: What participants will do (duration, activities)
  3. Risks: Any foreseeable risks or discomforts
  4. Benefits: Potential benefits to participant and science
  5. Confidentiality: How data will be protected
  6. Voluntary Participation: Right to withdraw without penalty
  7. Contact Information: Researcher and IRB contact details

4.3 Consent Documentation

Database Schema - Signup Model: - consent_text_version: TextField (full text of consent at time of signup) - consented_at: DateTimeField (timestamp of consent acceptance) - participant: ForeignKey (links to participant who consented) - status: CharField (tracks participation status)

Audit Trail: Every signup record permanently stores the exact consent form text shown to the participant, ensuring an immutable record of what was agreed to, even if the study consent is later updated.

5. Participant Protections

5.1 Voluntary Participation

  • Participants can cancel bookings before the cancellation window closes (default: 24 hours before session)
  • No penalties for cancellation within the allowed window
  • Alternative options available for fulfilling course credit requirements
  • Can withdraw from a study even after attending (no credit impact if IRB-approved)

5.2 No-Show Policy

Balanced Accountability:

The system tracks no-shows to encourage responsibility while protecting participant rights:

  • No-shows are recorded but do not automatically penalize participants
  • Excessive no-shows may trigger advisor notification (institution-specific threshold)
  • No-show data is not shared with researchers
  • Used only for administrative purposes

5.3 Credit Allocation Fairness

  • Transparent System: Credit values clearly displayed before signup
  • Automated Awards: Credits awarded automatically upon verified attendance (no researcher discretion)
  • Audit Trail: All credit transactions logged with timestamp and issuer
  • Appeals Process: Students can dispute credit awards through instructor

5.4 Data Rights

Participants have the right to:

  • Access: View all their own data stored in the system
  • Correction: Request correction of inaccurate profile information
  • Deletion: Request deletion of account and associated data (subject to record retention policies)
  • Export: Receive copy of their data in machine-readable format

6. IRB Compliance Features

6.1 IRB Approval Tracking

Field Purpose Required
IRB Status Approved, Exempt, Not Required, Pending, Expired Yes
IRB Protocol Number Official IRB tracking number If approved/exempt
IRB Expiration Date Approval expiration (automatic notifications) If approved/exempt
Consent Form Text Complete IRB-approved consent language Yes

6.2 Study Approval Workflow

  1. Researcher Submits Study: Creates study with IRB documentation
  2. Administrator Review: System administrator verifies IRB approval
  3. Approval Flag: Study marked as "approved" to appear to participants
  4. Expiration Monitoring: System tracks IRB expiration dates
  5. Automatic Deactivation: Studies automatically hidden if IRB approval expires

Automatic Compliance: The system prevents participants from signing up for studies with expired IRB approval by automatically hiding such studies from the public listing.

6.3 Audit Capabilities

The system maintains comprehensive audit logs for IRB review:

Audit Event Information Logged
Study Creation Researcher, timestamp, IRB status, all study parameters
Study Modification Changed fields, previous values, modifier, timestamp
Consent Agreement Participant, timestamp, consent text version, IP address
Credit Transaction Participant, amount, reason, issuer, timestamp
Attendance Marking Participant, status change, researcher, timestamp
Data Access User, accessed resource, timestamp, action

7. Data Management & Retention

7.1 Data Retention Policy

Data Type Retention Period Rationale
User Accounts (Active) Duration of enrollment + 1 year Support credit verification, appeals
Signup Records 5 years after study completion IRB record retention requirements
Credit Transactions 7 years (academic records) FERPA compliance
Research Data Per study-specific IRB protocol Researcher responsibility
Audit Logs 7 years Compliance verification
Consent Forms 3 years after study completion (minimum) Federal regulations (45 CFR 46)

7.2 Data Export and Portability

Researchers can export data in multiple formats:

  • CSV: For statistical analysis in SPSS, R, Excel
  • JSON: For programmatic access and complex data structures
  • PDF: For reports and documentation

7.3 Data Backup and Recovery

  • Daily Automated Backups: Full database backup every 24 hours
  • Point-in-Time Recovery: Can restore to any point within last 30 days
  • Geographic Redundancy: Backups stored in separate physical location
  • Encryption: All backups encrypted at rest
  • Tested Recovery: Regular restore testing to verify backup integrity

9. Open Science Framework Integration

9.1 Transparency and Preregistration

The system supports integration with the Open Science Framework (OSF):

  • Study Preregistration: Link to OSF preregistration
  • Materials Sharing: Public access to study materials
  • Data Sharing: Optional de-identified data publication
  • Protocol Transparency: Public protocol documentation

Optional Feature

OSF integration is optional and researcher-controlled. Studies can be run with or without OSF linking based on individual study needs and IRB requirements.

9.2 Bayesian Sequential Monitoring

For studies using sequential analysis:

  • Automated Bayes Factor calculation at specified intervals
  • Researcher notification when evidence threshold reached
  • Ethical early stopping when conclusions clear
  • Reduces unnecessary participant burden
  • All monitoring parameters preregistered
